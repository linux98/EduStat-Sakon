# เอกสารแผนและการจำลองการทดสอบระบบ (User Acceptance Testing - UAT Report)
**โครงการ:** ระบบสถิติสารสนเทศเพื่อการตัดสินใจด้านการศึกษา จังหวัดสกลนคร (EduData Sakon)
**เวอร์ชันปัจจุบัน:** v2.63 (Apps Script Version 162)
**วัตถุประสงค์:** เพื่อใช้เป็นคู่มือทดสอบและแบบประเมินผลการรับรองระบบสำหรับผู้ดูแลระบบและเจ้าหน้าที่ก่อนเปิดใช้งานอย่างเป็นทางการ

---

## 👥 1. สิทธิ์การเข้าใช้งานและข้อมูลบัญชีสำหรับทดสอบ (UAT Test Roles)

ระบบ EduData Sakon ใช้การบริหารความปลอดภัยและหน้าจอควบคุมแยกตาม **บทบาท (Role-based Access Control - RBAC)** โดยมีสิทธิ์แนะนำสำหรับทีมทดสอบ UAT ดังนี้:

| ลำดับ | บทบาท (Role) | สิทธิ์เมนูที่มองเห็น | จุดประสงค์หลักในการ UAT |
| :--- | :--- | :--- | :--- |
| 1 | **Super Admin / Admin** | แดชบอร์ดผู้บริหาร, กำกับติดตามการรายงาน, จัดการผู้ใช้งาน, ตั้งค่าระบบ, ประวัติการส่งรายงาน | ทดสอบการอนุมัติ/ส่งกลับ, จัดการบัญชีผู้ใช้, แก้ไข Deadline, แชตวิเคราะห์ด้วย AI BI |
| 2 | **Executive (ผู้บริหาร)** | แดชบอร์ดผู้บริหาร, ประวัติการส่งรายงาน | ทดสอบการดูภาพรวมสถิติจังหวัด, จำลองแผนนโยบาย (Policy Simulator) |
| 3 | **Agency User (เจ้าหน้าที่สังกัด)** | แดชบอร์ดผู้สังกัด (ใหม่), รายงานข้อมูลสถิติ (มีปุ่มกะพริบ), ประวัติการรายงาน | ทดสอบการเลือกแบบฟอร์ม, การกรอกตาราง, การอัปโหลด Excel/CSV, การดูประวัติรายงานของตนเอง |

---

## 🎬 2. สถานการณ์จำลองการทดสอบ 5 กลุ่มงานหลัก (Core Test Scenarios)

---

### 🟢 Scenario 1: การรายงานข้อมูลของเจ้าหน้าที่สังกัด (Data Reporting Flow)
*บทบาทผู้ทดสอบ: **Agency User (จนท. สังกัด เช่น จนท. สพม.สกลนคร - OBEC_M, พมจ. - MSDHS, อาชีวะ - VEC)***

#### **ขั้นตอนการทดสอบ (Test Steps):**
1. **เข้าสู่ระบบ:** ล็อกอินด้วยบัญชีสิทธิ์สังกัด (เช่น สพม.สกลนคร)
2. **สังเกตแถบเมนูด้านซ้าย:**
   * ไอคอนเมนู **"รายงานข้อมูลสถิติ"** ต้องมี **จุดกะพริบสีแดงเรืองแสง (Pulsing Red Dot) 🔴** แสดงจังหวะหายใจ (Breathing Glow) และไอคอนรูปเอกสารจะสเกลขยายขนาดเป็นจังหวะช้าๆ
3. **การเข้าหน้าจอกรอกรายงาน:**
   * คลิกที่เมนู "รายงานข้อมูลสถิติ" จุดสีแดงจะหายไป (เนื่องจากผู้ใช้เข้าหน้างานแล้ว)
   * **กรณีตารางขนาดใหญ่ (Multi-row):** เลือกแบบฟอร์มที่มีตาราง เช่น **"สถิติจำนวนนักเรียน ม.1 - ม.6"** 
     * **สิ่งที่ต้องเกิดขึ้น (UAT Pass):** สไลด์บาร์เมนูด้านซ้ายจะยุบพับตัวลงโดยอัตโนมัติ และแถบตารางจะขยายความกว้างเต็มสายตา 100%
     * **สิ่งที่ต้องเกิดขึ้น (UAT Pass):** จะปรากฏป้ายเรืองแสงสไตล์โมเดิร์นสีฟ้าด้านบนสุดเขียนว่า **`ขยายเมนูข้าง ➔`** ข้างปุ่ม ☰
     * ทดลองเอาเมาส์คลิกป้ายสีฟ้า **`ขยายเมนูข้าง ➔`** หรือปุ่ม ☰ แถบเมนูด้านซ้ายจะขยายกลับมาเต็มจอเหมือนเดิม
4. **ทดสอบระบบร่างเอกสารอัตโนมัติ (Auto-Save Draft):**
   * ทดลองพิมพ์ข้อมูลตัวเลขลงในช่องตาราง 2-3 ช่อง จากนั้นกดปิดเว็บเบราว์เซอร์ หรือกด Refresh หน้าจอ (F5)
   * **สิ่งที่ต้องเกิดขึ้น (UAT Pass):** ข้อมูลที่พิมพ์ค้างไว้ต้องไม่หายไป โดยระบบจะดึงแบบร่างในเบราว์เซอร์ขึ้นมาโหลดซ้ำลงตารางอัตโนมัติ
5. **ทดสอบคำสั่ง คัดลอก-วาง จาก Excel (Copy-Paste Integration):**
   * เปิดโปรแกรม Excel ในคอมพิวเตอร์ ทำการคลุมช่วงข้อมูลที่เป็นตัวเลขสถิติ (เช่น ขนาด 5 แถว 4 คอลัมน์) กด `Ctrl + C`
   * กลับมาที่ระบบ ดับเบิลคลิกเลือกช่องแรกในตารางกรอกข้อมูล แล้วกด `Ctrl + V`
   * **สิ่งที่ต้องเกิดขึ้น (UAT Pass):** ตัวเลขทั้งหมดจาก Excel จะต้องกระจายลงช่องตารางถัดไปทางขวาและแถวล่างให้อัตโนมัติอย่างถูกต้อง
6. **ทดสอบนำเข้าด้วยไฟล์แม่แบบ Excel/CSV (Import Engine):**
   * กดปุ่ม "เทมเพลต CSV" เพื่อรับไฟล์โครงสร้างคอลัมน์มาตรฐาน
   * ทดลองกดปุ่ม **"นำเข้าข้อมูลด้วย CSV"** หรือ **"นำเข้าข้อมูลจาก Excel"** แล้วเลือกไฟล์สถิติตามสังกัด (เช่น ชีต `'นร.พิการเรียนรวม'` สำหรับฟอร์ม `F14` หรือ ชีต `'N-net'` ของ สกร. เป็นต้น)
   * **สิ่งที่ต้องเกิดขึ้น (UAT Pass):** ข้อมูลจากไฟล์จะเข้ามาเติมในช่องตารางกรอกข้อมูลให้อัตโนมัติ และแสดงสถานะอัปโหลดสำเร็จ
7. **ส่งรายงาน:** กดปุ่ม **"บันทึกและส่งข้อมูล"** สังเกตป้ายแจ้งเตือนยืนยัน

### 🔵 Scenario 2: การตรวจสอบและรับรองข้อมูลของผู้ดูแลระบบ/ผู้บริหาร (Approval & Tracking Flow)
*บทบาทผู้ทดสอบ: **Super Admin / Admin***

#### **ขั้นตอนการทดสอบ (Test Steps):**
1. **เปิดหน้ากำกับติดตาม:** คลิกเมนู **"กำกับติดตามการรายงาน"**
   * **สิ่งที่ต้องเกิดขึ้น (UAT Pass):** จุดสีแดงจะขึ้นบอกยอดฉบับค้างตรวจสอบ โดยจุดสีแดงจะขยับไปอยู่มุมขวาบนของไอคอนรายงานเมื่อยุบแถบเมนู
2. **การคัดกรองข้อมูล:** ทดลองใช้แถบดร็อปดาวน์ด้านบนกรองเลือกปีการศึกษา และกรองสังกัด (เช่น เลือกดูเฉพาะสังกัด *สพม.สกลนคร*)
3. **การทดสอบหน้าต่างรายละเอียดเต็มจอ (Fullscreen Details Modal):**
   * ค้นหารายการแบบฟอร์มจำนวนนักเรียนของสังกัดที่ส่งข้อมูลเข้ามาแล้ว คลิกเพื่อเปิดหน้าต่างรายละเอียดข้อมูล
   * **สิ่งที่ต้องเกิดขึ้น (UAT Pass):** หน้าต่างโมดอลการ์ดจะขยายกว้างเกือบเต็มจอแสดงข้อมูลดิบเป็นตารางเรียงรายตามรายชื่อโรงเรียน
   * **สิ่งที่ต้องเกิดขึ้น (UAT Pass):** คอลัมน์ **"ชื่อสถานศึกษา"** และ **"ตำบล"** จะต้องแสดงในตำแหน่งแรกๆ ทางฝั่งซ้ายของตารางเสมอ (ไม่โผล่ไปอยู่ตรงกลางตารางสลับลำดับแบบสุ่ม)
   * **สิ่งที่ต้องเกิดขึ้น (UAT Pass) *จุดปรับปรุงล่าสุด*:** คอลัมน์สรุปตัวเลข **"นักเรียนทั้งหมด"** และ **"ห้องเรียนทั้งหมด"** จะต้องไม่ถูกนำไปจัดกลุ่มอยู่ใต้คำว่า "ชั้น ม.6" โดยจะต้องแยกคอลัมน์ของ ม.6 สถิติสิ้นสุดที่ ชาย-หญิง และคอลัมน์สรุปยอดรวมจะวางเป็นแถวคู่เดี่ยว (Rowspan 2) อยู่ขวาสุดอย่างสวยงาม
4. **ทดสอบการอนุมัติและส่งกลับแก้ไข:**
   * ลองกดปุ่ม **"อนุมัติรายงาน"** หรือปุ่ม **"ส่งกลับให้แก้ไข"** (พร้อมระบุเหตุผล เช่น *ขอให้ตรวจสอบข้อมูลจำนวนนักเรียน ม.2 อีกครั้ง*)
   * **สิ่งที่ต้องเกิดขึ้น (UAT Pass):** สถานะรายงานเปลี่ยนไปทันที และระบบจะบันทึกประวัติความเห็นและผู้รับรองลงฐานข้อมูลเบื้องหลัง

### 🟡 Scenario 3: การประมวลผลข้อมูลแดชบอร์ด (Dashboard Flow)
*บทบาทผู้ทดสอบ: **Executive User (ผู้บริหาร) และ Agency User (เจ้าหน้าที่สังกัด)***

#### **ขั้นตอนการทดสอบ (Test Steps) - สิทธิ์ผู้บริหาร/ส่วนกลาง:**
1. **ความเร็วการประมวลผล (10x Speedup):** คลิกสลับเข้าเมนู **"แดชบอร์ดผู้บริหาร"**
   * **สิ่งที่ต้องเกิดขึ้น (UAT Pass):** แดชบอร์ดภาพรวมทั้งหมด รวมถึงแผนที่และกราฟ ECharts สถิติจำนวนรวมของจังหวัด จะต้องประมวลผลเสร็จและแสดงผลรวดเร็วภายใน **0.6 - 0.8 วินาที** (ไม่มีอาการหน้าขาวค้างรอ 10 วินาทีแบบเดิม)
2. **การฟิลเตอร์ข้อมูล:** ทดลองคลิกที่พื้นที่กราฟวงกลม หรือเปลี่ยนดร็อปดาวน์ปีการศึกษา/สังกัด ตัวเลขแดชบอร์ดจะอัปเดตแบบเรียลไทม์

#### **ขั้นตอนการทดสอบ (Test Steps) - สิทธิ์สังกัด (Agency):**
1. **การแสดงผลแดชบอร์ดสะสมของสังกัด:** ล็อกอินด้วยสิทธิ์เจ้าหน้าที่สังกัด เข้าหน้าแดชบอร์ด
   * **สิ่งที่ต้องเกิดขึ้น (UAT Pass) *จุดปรับปรุงล่าสุด*:** จะมีกรอบการ์ดสถิติสรุปยอดสะสม **"📊 สรุปข้อมูลสถิติล่าสุดประจำสังกัด (ที่อนุมัติแล้ว)"** เพิ่มขึ้นมา แสดงตัวเลขอภิมหาข้อมูลสะสมจริงของสังกัด ได้แก่ จำนวนโรงเรียน, ยอดผู้เรียน, จำนวนครู, และนักเรียนพิเศษ
2. **กราฟแนวโน้มเปรียบเทียบกับปีก่อน (YoY):**
   * สังเกตกราฟแท่งเปรียบเทียบข้อมูลย้อนหลัง 2 ปีการศึกษา
   * **สิ่งที่ต้องเกิดขึ้น (UAT Pass) *จุดปรับปรุงล่าสุด*:** ข้อมูลแท่งกราฟจะดึงจากประวัติสถิติสะสมจริงของสังกัดในปีที่ผ่านมา (ไม่ได้นำค่าปัจจุบันมาหักลบ 5% แบบ Mock อีกต่อไป)
   * **สิ่งที่ต้องเกิดขึ้น (UAT Pass) *จุดปรับปรุงล่าสุด*:** ตัวกราฟแท่ง YoY และการ์ดแสดงผลขวาจะวางตัวอยู่ในหน้าจออย่างเป็นระเบียบ **ขอบขวาไม่ถูกบีบล้นออกนอกจอ** และไม่มีแถบเลื่อนแนวนอนกวนใจ แม้ว่าผู้ใช้จะทำการยืดขยายแถบเมนูข้างเข้าออกก็ตาม
3. **การทดสอบ Live Sync:**
   * สับสวิตช์มุมขวาบน **"Live Sync"**
   * **สิ่งที่ต้องเกิดขึ้น (UAT Pass):** จุดกะพริบเปลี่ยนเป็น **สีเขียวเรืองแสง 🟢** พร้อมเวลานับถอยหลังจะลดลงทีละวินาที เมื่อครบ 5 นาทีจะอัปเดตดึงข้อมูลหลังบ้านโดยอัตโนมัติ

### 🔵 Scenario 4: ระบบ AI สรุปวิเคราะห์และ Policy Simulator (AI BI Features)
*บทบาทผู้ทดสอบ: **Super Admin / Executive User***

#### **ขั้นตอนการทดสอบ (Test Steps):**
1. **ฟีเจอร์ AI สรุป (Policy Summary):** กดปุ่ม **"AI สรุป"** ด้านขวาบน
   * **สิ่งที่ต้องเกิดขึ้น (UAT Pass):** ระบบจะเรียกข้อมูลหลังบ้านทั้งหมดส่งไปให้ AI ประมวลผลเชิงวิเคราะห์นโยบายและส่งกลับบทสรุปเชิงลึกมาแสดง
2. **ระบบจำลองแผนนโยบาย (Policy Simulator):**
   * เปิดสวิตช์ **"เปิดโหมดจำลอง"** บนแถบการ์ดจำลองนโยบายสีฟ้า
   * ทดลองขยับแถบเลื่อน (Sliders) เช่น ปรับเพิ่ม "อัตราการเติบโตผู้เรียน" หรือปรับลด "อัตราการเติบโตครู"
   * **สิ่งที่ต้องเกิดขึ้น (UAT Pass):** ตัวเลขคาดการณ์อัตราส่วนครูต่อนักเรียน (Ratio) ด้านล่างสุดจะแปรเปลี่ยนคำนวณใหม่ตามสไลเดอร์ทันที
   * กดปุ่ม **"AI คาดการณ์ 3 ปีล่วงหน้า"** เพื่อเรียกอ่านแผนพยากรณ์โครงสร้างประชากรผู้เรียนที่จำลองตามสไลเดอร์
3. **ระบบตอบคำถามข้อมูลสถิติ (Gemini BI Chat):**
   * กดปุ่มฟังก์ชันเรียกแชตวิเคราะห์กับ AI (Gemini BI)
   * พิมพ์คำถามยากๆ เช่น *"ช่วยหาโรงเรียนในสังกัดที่มีอัตราส่วนครูต่อนักเรียนน้อยที่สุดให้หน่อย"* หรือ *"เปรียบเทียบสถิติจำนวนนักศึกษา มรภ.สกลนคร กับ มก.สกลนคร"*
   * **สิ่งที่ต้องเกิดขึ้น (UAT Pass) *จุดปรับปรุงก่อนหน้า*:** AI จะต้องวิเคราะห์และนำส่งผลลัพธ์ได้อย่างแม่นยำรวดเร็วโดยไม่มีอาการเกิด Error แจ้งเตือน JSON Parse ขัดข้อง (อันเนื่องมาจากปัญหาอัญประกาศคู่ตัวแปรในชื่อและเนื้อความ)

### 🟢 Scenario 5: ระบบตั้งค่าและบริหารบัญชี (Management & Settings)
*บทบาทผู้ทดสอบ: **Super Admin / Admin***

#### **ขั้นตอนการทดสอบ (Test Steps):**
1. **การจัดการผู้ใช้ (User Management):** เข้าเมนู **"จัดการผู้ใช้งาน"**
   * ทดลองสร้างบัญชีใหม่, แก้ไขข้อมูลชื่อ, ตั้งค่าเปลี่ยนรหัสผ่าน, กดระงับบัญชี (Suspend) และกดลบ (Delete)
   * **สิ่งที่ต้องเกิดขึ้น (UAT Pass):** ระบบจะเขียนข้อมูลการอัปเดตตารางกลับลงฐานข้อมูลชีต `Users` ได้ทันทีและส่งผลต่อการล็อกอินจริงของผู้ใช้งานรายนั้น
2. **การตั้งค่าวันครบกำหนด (Deadline Settings):** เข้าเมนู **"ตั้งค่าระบบ"**
   * ปรับตั้งค่าวันส่งแบบฟอร์มกำหนดส่งรายงานของสังกัดต่างๆ
   * **สิ่งที่ต้องเกิดขึ้น (UAT Pass):** วันส่งผลลัพธ์ของแบบฟอร์มเป้าหมายของฝั่งเจ้าหน้าที่รายงานในแถบการกรอกจะเปลี่ยนอัปเดตตามทันที

## 💡 3. คำแนะนำเพิ่มเติมจากผู้พัฒนาเพื่อประสิทธิภาพสูงสุด (Developer Recommendations)

จากการพัฒนาและตรวจระบบโดยละเอียด ผมมีข้อแนะนำสำคัญ 4 ประการเพื่อให้ระบบทำงานได้อย่างเสถียรและทนทานในระยะยาวครับ:

### 3.1. 🔑 การจัดการคีย์เชื่อมต่อระบบภายนอก (API Key Management)
* **การติดตั้ง:** คีย์ API สำหรับประมวลผล Gemini AI BI ถูกบันทึกไว้ใน **Script Properties** (`gemini_api_key`) ในระบบหลังบ้าน Google Apps Script เรียบร้อยแล้ว ซึ่งปลอดภัยจากการถูกโจรกรรมข้อมูลผ่านหน้าบ้าน
* **คำแนะนำ:** ควรตั้งเวลาตรวจสอบหรือสับเปลี่ยนคีย์ (API Key Rotation) ทุกๆ 6-12 เดือนบน Google Cloud Console เพื่อป้องกันโควตาจำกัดกรณีมีการส่งคำขอปริมาณมหาศาลพร้อมกันในช่วงสิ้นงวดการส่งรายงาน

### 3.2. 📂 การรักษาระเบียบโครงสร้างหัวไฟล์ตาราง Excel นำเข้า
* **ตรรกะระบบ:** พาร์สเซอร์ผู้อ่านไฟล์ Excel ของหน้าบ้านเขียนขึ้นโดยใช้การแปลงหัวภาษาไทยในแถวแรกสุด (Header Rows) ไปเป็นฟิลด์ฐานข้อมูล
* **คำแนะนำ:** หากสังกัดมีการดาวน์โหลดแบบฟอร์มโครงสร้างใหม่เพื่อทำไฟล์นำเข้าภายนอก **ห้ามแก้ไขตัวสะกดในแถวหัวตารางเด็ดขาด** (เช่น หัวข้อคอลัมน์ชื่อโรงเรียน หรือระดับชั้นเรียน) เพราะหากคอลัมน์ไม่ตรงระบบพาร์สเซอร์จะข้ามการแปลงข้อมูลช่องนั้นเพื่อป้องกันข้อมูลเน่าเสีย

### 3.3. 🗃️ การบริหารจัดการพื้นที่เก็บข้อมูล Google Sheets
* **ขีดจำกัด:** Google Sheets รองรับปริมาณเซลล์สูงสุดที่ **10,000,000 เซลล์** ต่อหนึ่งสเปรดชีต
* **คำแนะนำ:** แม้ว่าระบบปัจจุบันจะมีขนาดกะทัดรัดและประมวลผลได้รวดเร็ว (ต่ำกว่า 1 วินาที) แต่หากสะสมประวัติข้อมูลสถิติรายงานไปมากกว่า 5-10 ปีการศึกษา แนะนำให้ผู้ดูแลระบบกด **"ดาวน์โหลดเก็บไฟล์ข้อมูลสถิติของปีเก่าๆ สำรองแยกชีต"** แล้วกดล้างแถวของปีการศึกษาอดีต เพื่อคงขนาดชีตใช้งานให้เบาบางและตอบสนองได้เร็วที่สุดตลอดไป

### 3.4. 🔄 การเคลียร์แคชข้อมูลผู้บริหารหลังเริ่มปีการศึกษาใหม่
* **สถาปัตยกรรม:** ระบบใช้ CacheService ของ Apps Script (เก็บข้อมูล 6 ชั่วโมง) เพื่อรักษาความเร็วระดับสุดยอดในการแสดงผลหน้าแดชบอร์ดหลัก
* **คำแนะนำ:** เมื่อเปลี่ยนผ่านเข้าสู่ปีการศึกษาถัดไปอย่างเป็นทางการ ผู้ดูแลระบบสามารถคลิกปุ่ม **"ซิงค์ข้อมูลล่าสุด" (Sync Button)** ที่หัวขวาของกระดานผู้บริหาร หรือใช้ปุ่มล้างแคช เพื่อสั่งให้ระบบทำลายแคชเกลี้ยงเกลาและสร้างสถิติประมวลผลของปีใหม่ให้สดใหม่ที่สุดในทันที
